# Validation Checks

## Purpose of this file
This file confirms whether the Instrumentation + Automation + Capstone lab is complete and correct.

The goal is not only to verify that a dashboard exists, but to confirm that the learner understands how Azure DevOps supports visibility, notification, event-driven response, and complete workflow interpretation.

## Validation approach
For this lab, validation is based on:
- dashboard usefulness
- signal relevance
- notification understanding
- service-hook or webhook understanding
- capstone explanation quality
- learner operational thinking

## Check 1 — A visibility surface exists and is meaningful
Confirm that:
- a dashboard exists or has been refined meaningfully
- the dashboard name is clear
- the learner can explain the purpose of the dashboard

This proves that the project now has an operational summary view.

## Check 2 — The chosen widgets or views answer real questions
Confirm that:
- the widgets included are relevant
- each one supports a useful operational or delivery question
- the dashboard is readable and not overloaded

This proves that the learner is thinking in terms of signal value, not only platform features.

## Check 3 — Analytics or trend thinking is present
Confirm that:
- the learner understands that trends matter, not only point-in-time status
- the learner can explain how reporting or analytics-backed widgets help teams
- the learner has moved beyond pure execution into visibility thinking

This proves that the operational view is maturing.

## Check 4 — Notification purpose is clear
Confirm that:
- the learner understands at least one useful notification scenario
- the learner can explain why that event deserves attention
- notifications are being treated as operational awareness tools

This proves that the response layer is present.

## Check 5 — Service-hook or webhook automation is understood
Confirm that:
- the learner understands that Azure DevOps events can be sent outward
- the learner can describe one event-driven automation story
- the learner understands the difference between internal visibility and external event-driven automation

This proves that the platform is being understood as integration-ready, not isolated.

## Check 6 — The dashboard helps the learner explain the system
Confirm that:
- the learner can use the visibility surface to talk about project health
- the learner can point to work, build, or delivery signals in a coherent way
- the dashboard supports explanation, not only observation

This proves that the visibility layer is actually useful.

## Check 7 — The full foundation-phase workflow can be explained
Confirm that the learner can explain:
- how work is planned
- how code is managed
- how CI validates changes
- how CD promotes artifacts
- how packages support reuse
- how security protects delivery
- how instrumentation and automation make the workflow visible and manageable

This is the capstone validation for the entire first phase.

## Check 8 — You can explain why this module matters in your own words
This is the most important validation for this module.

You should now be able to explain:
- why visibility matters
- why dashboards should stay focused
- why notifications should be useful
- why service hooks enable event-driven automation
- why this module completes the Azure DevOps foundation phase

If you can explain this clearly, the module is truly understood.

## Expected success indicators
The strongest success signals for this lab are:
- the dashboard is meaningful and clear
- the learner understands which signals matter
- notification and event-driven automation thinking are present
- the full Azure DevOps foundation story is coherent
- the learner feels ready to move into AKS delivery engineering

## If validation fails
If any validation step fails:
- simplify the dashboard
- reduce low-value signals
- revisit the purpose of notifications
- re-check the service-hook story
- strengthen the capstone explanation before moving forward
