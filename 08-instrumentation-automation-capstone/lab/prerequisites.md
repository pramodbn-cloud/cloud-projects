# Prerequisites

## Purpose of this file
This file defines everything that must already be ready before starting the Instrumentation + Automation + Capstone lab.

This module depends on the learner already having a meaningful Azure DevOps foundation workflow to observe and summarize.

## Required account access
You should have working access to:
- Azure DevOps
- the organization used in earlier modules
- the project used for this learning repository
- dashboards, widgets, and reporting-related areas
- notifications and service-hook areas if available in your role

## Required tools
The following should be available:

- a web browser for Azure DevOps access
- the working project and pipeline setup from earlier modules
- optional note-taking for recording useful signals and capstone conclusions

No heavy local tooling is required for this module.

## Required permissions
You should be able to:
- view or edit dashboards
- add or configure widgets if allowed
- review notifications if allowed
- review or configure service hooks if allowed
- inspect pipeline, work-item, and project activity

If working in a restricted environment, it is still acceptable to model some automation steps conceptually.

## Required prior modules
You should complete:
- Folder 01 — Azure DevOps Foundation Setup
- Folder 02 — Agile Planning + Traceability
- Folder 03 — Source Control Strategy
- Folder 04 — CI Pipelines: Build + Test + Artifacts
- Folder 05 — CD Pipelines: Deployments + Approvals + Rollback
- Folder 06 — Azure Artifacts: Package Management Strategy
- Folder 07 — Security + Compliance: DevSecOps Basics

This module works best when the learner already has a complete foundation-phase workflow to reflect on.

## Required environment state
Before starting:
- the project should contain planning, repo, and pipeline activity from earlier modules
- the learner should understand the difference between execution and visibility
- dashboards or reporting features should be accessible
- the learner should be ready to think about signal value and operational awareness

## Visibility scenario recommendation
Keep the lab simple and meaningful.

Recommended focus areas:
- one dashboard with a few useful widgets
- one notification or alerting example
- one service-hook or webhook automation example, real or conceptual
- one capstone summary of the full workflow

## Naming convention to follow
Use clear and professional names.

Recommended examples:
- dashboard name: `Foundation Delivery Overview`
- dashboard name: `Azure DevOps Job Ready Capstone`
- service hook label: aligned to the event and destination
- notification names that reflect their purpose clearly

Keep names readable so later modules can reference them if needed.

## Definition of ready
The learner is ready to start this lab when:
- earlier foundation modules are completed
- dashboards and visibility surfaces are accessible
- the learner is ready to decide which delivery signals actually matter
- the learner understands that this module is both operational and reflective
- the learner is prepared to summarize the full Azure DevOps foundation journey
