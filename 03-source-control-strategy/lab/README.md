# Lab Guide — Source Control Strategy

## Lab objective
The objective of this lab is to help the learner create a clean and professional source control workflow inside Azure DevOps using Azure Repos.

This lab focuses on repository creation, local connection, basic branch workflow, commit discipline, and pull request understanding. The goal is not to create a complex application, but to make the learner comfortable with the real operational flow of how code is introduced and merged safely.

## What you will build or configure
In this lab, you will create or configure a repository, connect it to your local machine, add starter content, create a branch, make a change, push it, and create a pull request.

This creates a full beginner-to-professional source control cycle inside Azure DevOps and prepares the learner for build automation in later modules.

## Why this lab matters
Planning defines what should be done, but source control defines how implementation enters the shared engineering system.

This lab matters because clean source control habits are what allow teams to build safely, collaborate effectively, review changes intelligently, and connect code activity back to the delivery workflow.

## Expected final outcome
By the end of this lab, the learner should have:
- a working repository in Azure DevOps
- a connected local working copy
- a simple branch-based change flow
- a pull request created and reviewed
- a strong understanding of safe collaboration in source control

## Skills gained
- Creating and using repositories in Azure DevOps
- Cloning and working locally with version-controlled content
- Creating and using branches for isolated changes
- Understanding commits, push flow, and pull request collaboration

## Real-world relevance
In real engineering teams, repositories are where implementation becomes visible, shareable, reviewable, and recoverable. Branches protect the main code line, pull requests improve collaboration quality, and commit history preserves how the system evolved over time.

This lab reflects those same real-world collaboration patterns.

## Completion standard
A learner should not mark this lab complete until:
- the repository exists and is usable
- the local machine is connected to it
- a feature branch workflow has been completed
- a pull request has been created and understood
- validation checks confirm the full flow worked correctly
