# Prerequisites

## Purpose of this file
This file defines everything that must already be ready before starting the Agile Planning + Traceability lab.

This module is conceptually simple, but it depends on the Azure DevOps environment already being set up properly.

## Required account access
You should have working access to:
- Azure DevOps
- the organization created or confirmed in Folder 01
- the project that will be used for the full learning journey

## Required tools
For this module, the main tool is the Azure DevOps web interface.

Recommended access:
- modern web browser
- stable internet connection
- optional note-taking tool for recording the work hierarchy you design

You do not need advanced CLI tools for this lab.

## Required permissions
You should be able to:
- open Azure Boards
- create work items
- edit work items
- create or assign iteration paths if allowed in your environment
- view dashboards, boards, and backlog structure

If working in a restricted organizational environment, confirm that work item editing permissions are available.

## Required prior modules
You should complete:
- Folder 01 — Azure DevOps Foundation Setup

That module created the platform foundation required for this planning module.

## Required environment state
Before starting:
- the Azure DevOps project should exist
- you should be able to open the project successfully
- Azure Boards should be visible in the project navigation
- the learner should understand the basic purpose of the platform

## Planning scenario recommendation
Before starting implementation, choose a simple practice scenario.

Recommended options:
- building a small web application
- setting up a CI/CD starter workflow
- creating an internal team dashboard
- onboarding a basic AKS deployment flow

The purpose is not to create a perfect real backlog, but to create a realistic enough structure to understand the planning flow.

## Naming convention to follow
Use clear and readable work item names.

Recommended style:
- Epic: broad objective language
- Feature: meaningful capability-based names
- User Story: action or outcome oriented
- Task: implementation-level action language

Keep names simple, specific, and easy to understand later.

## Definition of ready
The learner is ready to start this lab when:
- Azure DevOps project access is working
- Azure Boards is visible
- work items can be created
- a simple delivery scenario has been chosen
- the learner is ready to think in hierarchy and traceability, not just ticket creation
