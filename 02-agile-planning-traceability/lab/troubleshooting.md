# Troubleshooting

## Purpose of this file
This file helps the learner understand common issues that appear during the Agile Planning + Traceability lab.

The most common problems in this module are not infrastructure problems. They are usually structure problems, naming problems, or understanding problems. That is why troubleshooting here is strongly connected to planning clarity.

---

## Common issue 1 — Work items feel random and disconnected

### Symptoms
- items were created, but they do not feel related
- hierarchy looks forced or unnatural
- learner is unsure what belongs where
- backlog looks like a list, not a structure

### Likely cause
The planning scenario was either too vague or too broad. Another common cause is creating work items before defining the main delivery objective clearly.

### Fix
- pause and restate the scenario in one sentence
- define the top-level objective first
- then define 2 to 3 major areas under it
- then create smaller outcome-oriented items under those areas
- remove or rename items that do not fit the structure

### Re-validation
After the fix:
- open the backlog again
- confirm that the hierarchy reads naturally from top to bottom
- confirm that each child item clearly supports its parent item

---

## Common issue 2 — Learner confuses features, stories, and tasks

### Symptoms
- tasks are written like big objectives
- stories are written like low-level actions
- the levels feel mixed up
- hierarchy exists but has weak meaning

### Likely cause
The learner is focusing too much on item types and not enough on planning intent.

### Fix
Think of the levels like this:
- top level = bigger goal
- middle level = meaningful delivery chunk
- lower level = outcome to be achieved
- task level = action needed to complete it

Use plain language first. The exact labels matter less than the planning logic.

### Re-validation
After the fix:
- review one branch of the hierarchy
- confirm the size and purpose of each level decreases logically as you move downward

---

## Common issue 3 — Board view and backlog view feel confusing

### Symptoms
- learner understands backlog but not board
- learner sees items but does not understand the difference between views
- the platform feels visually crowded

### Likely cause
The learner is expecting all views to explain the same thing in the same way.

### Fix
Remember:
- backlog view shows structured work and hierarchy
- board view shows movement and state
- sprint view emphasizes time-bound execution

These are not duplicate screens. They show different operational perspectives on the same work.

### Re-validation
After the fix:
- open backlog view and explain what it shows
- open board view and explain what changes there
- if you can clearly describe the difference, the confusion is resolved

---

## Common issue 4 — Iteration or sprint setup feels unclear

### Symptoms
- learner sees sprint options but does not know what to do with them
- iteration paths feel like configuration noise
- no clear connection is felt between sprint setup and work items

### Likely cause
The learner is trying to master sprint administration instead of understanding sprint intent.

### Fix
For this module, keep it simple:
- think of a sprint as a time-box for selected work
- associate only a small number of items if needed
- focus on understanding time-based grouping, not advanced project administration

### Re-validation
After the fix:
- confirm you can explain why some work belongs to a current sprint and some does not
- confirm you understand that iteration structure adds timing to planned work

---

## Common issue 5 — Traceability still feels abstract

### Symptoms
- learner completed the lab but still cannot explain traceability
- work items exist, but the reason for the structure is unclear
- the module feels administrative instead of meaningful

### Likely cause
The learner may be focusing too much on clicks and not enough on workflow meaning.

### Fix
Reframe traceability like this:
- a business or delivery goal starts the process
- that goal becomes organized work
- that work can be assigned and tracked
- later, code and delivery activities should connect back to it
- this connection helps teams explain progress and purpose

### Re-validation
After the fix:
- say the full workflow out loud in your own words
- if you can describe how planning will later connect to code and delivery, the concept is now clear

---

## Troubleshooting mindset
While debugging this module, always ask:
- is my scenario clear enough?
- does my hierarchy show logical reduction from big goal to execution work?
- am I confusing work item labels with work item purpose?
- do I understand the difference between backlog, board, and sprint views?
- am I treating traceability as workflow logic rather than just tool usage?

## Escalation rule
If the planning model still feels confusing:
1. simplify the scenario  
2. reduce the number of work items  
3. rebuild one clean hierarchy branch  
4. validate logic before adding more items  
5. focus on meaning, not volume
