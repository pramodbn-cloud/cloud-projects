# Implementation Steps

## Purpose of this file
This file contains the practical execution flow for the Agile Planning + Traceability lab.

The goal is to help the learner create a small but meaningful planning structure in Azure DevOps and understand how work becomes visible, organized, and trackable before technical execution begins.

---

## Step 1 — Understand the planning goal
Before creating work items, understand what this module is trying to teach.

This lab is not about creating many tickets. It is about learning how a delivery objective is shaped into a structured backlog that a team can actually use.

At the end of this lab, you should have:
- one simple planning scenario
- a clean work item hierarchy
- basic iteration or sprint alignment
- confidence in explaining traceability

## Step 2 — Open Azure Boards inside your project
Go to your Azure DevOps project and open the Boards area.

Spend a few minutes observing the available sections such as:
- work items
- backlog
- boards
- sprints
- queries

Do not configure everything at once. First become comfortable with the fact that this is the planning surface for delivery work.

## Step 3 — Define a small practice scenario
Choose one realistic learning scenario that you will use for this lab.

Example scenario:
“Set up a starter delivery workflow for a small web application using Azure DevOps.”

This scenario is only a learning anchor. It helps you avoid creating abstract work items that do not connect to a real delivery purpose.

## Step 4 — Create the top-level work item
Create the top-level item that represents the larger objective.

Depending on your process configuration, this may be an Epic or a Feature. The important thing is not only the item type, but the planning intention behind it.

Example:
- Epic: Build and deliver a starter internal application workflow

This creates the anchor for the rest of the hierarchy.

## Step 5 — Create the next level of breakdown
Under the top-level objective, create 2 to 3 meaningful work areas.

These should represent larger capability chunks, not tiny actions.

Examples:
- Planning and project setup
- Source control and collaboration setup
- Build and release workflow setup

At this stage, you are teaching yourself to think in structured delivery slices.

## Step 6 — Create implementation-level stories or requirements
Now create smaller items under each major work area that reflect actual outcomes the team would need.

Examples:
- As a team member, I want the project structure defined so that work can be tracked clearly
- As a developer, I want the code repository ready so that I can begin development
- As an engineer, I want a build pipeline so that code changes are validated automatically

These should still be outcome-oriented, not overly technical task fragments.

## Step 7 — Create supporting tasks
Now create a few tasks under one or more stories to represent execution-level steps.

Examples:
- Create project naming standard
- Define branch naming pattern
- Review pipeline trigger approach
- Assign initial work ownership

This is where the learner starts to see how strategic work becomes actionable work.

## Step 8 — Review the hierarchy
Pause and review the structure you created.

Ask:
- Does the top-level item clearly describe the main objective?
- Do the lower levels logically fit under it?
- Do the stories represent outcomes instead of vague labels?
- Do the tasks feel like actual execution work?

The purpose here is not quantity. It is structural quality.

## Step 9 — Set iteration or sprint context
Open the sprint or iteration-related area in Azure DevOps and review how time-based delivery can be applied.

If appropriate in your environment:
- create a simple iteration path
- associate selected work items with that iteration
- review how work appears in the sprint view

This step helps the learner understand that work is not only structured hierarchically, but also sequenced over time.

## Step 10 — Assign ownership and state where appropriate
For a few work items:
- assign an owner if useful
- set a basic state
- add a short description

This shows how work becomes actionable and trackable, not just listed.

## Step 11 — Open backlog and board views
Now open both the backlog view and board view and observe the same work from different angles.

Notice:
- backlog shows structured work
- board shows movement and execution flow

This is one of the most important learning moments in this module because it reveals that planning and execution visibility are connected.

## Step 12 — Create one simple query if available
Create a basic query to filter or view a useful slice of work.

For example:
- all active work items
- all work assigned to a user
- all work inside the current sprint

This helps the learner understand that work can be organized and inspected dynamically, not only manually.

## Step 13 — Reflect on traceability
Pause and think about what traceability means in practical terms.

In this module, traceability means:
- a large goal is visible
- it is broken into smaller work
- that work can be assigned, tracked, filtered, and later connected to implementation

This mindset is the real objective of the module.

---

## Checkpoint
At this point, the following should already be true:

- a realistic planning scenario exists
- work items are created in a meaningful hierarchy
- at least one iteration or sprint structure has been reviewed or applied
- work is visible in backlog and board views
- the learner understands the connection between structure and execution

## Step 14 — Prepare for source control thinking
Before moving to the next module, recognize the next workflow step.

The next module is about repositories, branching, and pull requests. When you reach that stage, the planning structure created here should help you think more clearly about what implementation work is tied to which backlog objective.

This is the bridge from planning to code collaboration.

---

## Implementation notes
- keep the scenario small and realistic
- avoid creating too many work items
- focus on logical structure, not volume
- use language that reflects outcomes, not random labels
- treat backlog hierarchy as a thinking tool, not an admin task

## End-of-implementation summary
In this lab, you:
- opened and explored Azure Boards
- selected a simple delivery scenario
- created a work item hierarchy
- reviewed sprint or iteration structure
- viewed work in backlog and board formats
- practiced the core idea of traceability from goal to execution

You are now ready to validate whether the planning setup is correct and understandable.
