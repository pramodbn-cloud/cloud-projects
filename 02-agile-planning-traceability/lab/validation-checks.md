# Validation Checks

## Purpose of this file
This file confirms whether the Agile Planning + Traceability lab is complete and correct.

The goal is not only to verify that work items were created, but to confirm that they were created in a way that teaches planning logic, hierarchy, and traceability.

## Validation approach
For this lab, validation is based on:
- work item structure verification
- planning logic verification
- sprint/iteration visibility
- backlog and board visibility
- learner explanation ability

## Check 1 — Work items exist in a meaningful hierarchy
Confirm that you created:
- a top-level objective
- a second layer of major work
- at least one deeper implementation-oriented layer
- optional tasks for execution detail

This proves that you understand structured planning rather than flat ticket creation.

## Check 2 — Work item names are clear and meaningful
Review the names of the items you created.

Confirm that:
- they are understandable
- they reflect actual delivery meaning
- they are not vague, random, or overly technical without context

This proves that your planning is readable and explainable.

## Check 3 — Backlog view reflects the structure properly
Open the backlog view and confirm that:
- the hierarchy is visible
- items appear under the expected parents
- the structure makes sense when read top to bottom

This proves that the work breakdown is usable.

## Check 4 — Board or sprint view provides execution visibility
Open the board or sprint-related view and confirm that:
- work is visible in a state-based or sprint-oriented way
- you can see how planning connects to execution tracking
- the view supports action, not only documentation

This proves that the planning structure can support delivery flow.

## Check 5 — Iteration or sprint logic is understood
If you configured an iteration or sprint, confirm that:
- selected work is associated correctly
- the learner understands why time-bounded organization matters
- the sprint view makes sense in relation to backlog planning

If full iteration configuration was not possible in your environment, you should still be able to explain the purpose of it clearly.

## Check 6 — Query or filtered view works if created
If you created a query, confirm that:
- it returns the expected work
- it helps you inspect a meaningful subset
- it demonstrates that Azure Boards can support focused tracking

This proves that the learner understands more than static item lists.

## Check 7 — You can explain traceability in your own words
This is the most important validation for this module.

You should now be able to explain:
- why work starts as a larger objective
- why it is broken into smaller items
- how those items support execution
- how boards, backlog, and sprints contribute to visibility
- why later technical work should connect back to this planning layer

If you can explain this well, the module is truly understood.

## Expected success indicators
The strongest success signals for this lab are:
- work item hierarchy exists and makes sense
- the backlog is readable
- board or sprint views show execution relevance
- the learner understands how work planning supports later delivery workflows

## If validation fails
If any validation step fails:
- revisit the implementation sequence
- simplify the planning scenario
- correct the hierarchy if it feels forced or confusing
- re-check work item relationships
- use `troubleshooting.md` for common planning issues
